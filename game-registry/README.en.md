# game-registry

`game-registry` is the service registration and discovery module of the game server framework. It provides a unified `Registry` / `Discovery` API and plugs registry implementations — Consul, Nacos, Etcd, Apache ZooKeeper — in via Java SPI.

## Module Structure

| Module | Description |
| --- | --- |
| `game-registry-core` | API, exceptions, factory, SPI and shared support utilities |
| `game-registry-zookeeper` | ZooKeeper implementation based on Apache Curator |
| `game-registry-etcd` | Etcd implementation based on jetcd |
| `game-registry-nacos` | Nacos implementation based on the Nacos Java Client |
| `game-registry-consul` | Consul implementation based on the Consul Java Client |
| `game-registry-starter` | Lightweight starter that pulls in only core by default; providers are added explicitly as needed |

## Design Principles

- Business code depends only on the core APIs: `Registry`, `Discovery`, `ServiceInstance`, etc.
- Registry implementations are mounted via the `RegistryProvider` SPI; adding an implementation requires no change to core. `game-registry` is only a consistency wrapper — it does not replace or restrict direct use of the native SDKs.
- `RegistryFactory` selects providers by string type key, while `RegistryType` remains as a compatibility entry for the built-in types.
- `RegistryBundle` exposes both the registration side and the discovery side; for the built-in providers both capabilities are currently implemented by the same object.
- Eureka is not among the built-in providers; for Eureka or other registries, extend via the SPI.

## Quick Start

### 1. Add dependencies

To use the starter's builder/factory entry points, add the starter plus the provider for your target registry:

```xml
<dependency>
    <groupId>com.github.huaye2007.mana</groupId>
    <artifactId>game-registry-starter</artifactId>
    <version>1.0.0-SNAPSHOT</version>
</dependency>
<dependency>
    <groupId>com.github.huaye2007.mana</groupId>
    <artifactId>game-registry-zookeeper</artifactId>
    <version>1.0.0-SNAPSHOT</version>
</dependency>
```

Without the starter, add just core and the provider:

```xml
<dependency>
    <groupId>com.github.huaye2007.mana</groupId>
    <artifactId>game-registry-core</artifactId>
    <version>1.0.0-SNAPSHOT</version>
</dependency>
<dependency>
    <groupId>com.github.huaye2007.mana</groupId>
    <artifactId>game-registry-zookeeper</artifactId>
    <version>1.0.0-SNAPSHOT</version>
</dependency>
```

### 2. Create the registry components

```java
RegistryConfig config = new RegistryConfig();
config.setType(RegistryType.ZOOKEEPER);
config.setEndpoints("127.0.0.1:2181");
config.setBasePath("/services");

RegistryBundle bundle = RegistryFactory.create(config);
Registry registry = bundle.getRegistry();
Discovery discovery = bundle.getDiscovery();

bundle.start();
```

The config can also be created via the builder:

```java
RegistryConfig config = RegistryConfig.builder()
        .type(RegistryType.ZOOKEEPER)
        .endpoints("127.0.0.1:2181")
        .basePath("/services")
        .property("zone", "shanghai-a")
        .build();
```

For built-in providers the registration side and the discovery side are currently the same object; third-party providers may return two objects. Prefer `RegistryBundle#start()` and `RegistryBundle#close()` to manage the lifecycle uniformly.

A string type also works, which is convenient for third-party providers:

```java
config.setType("zookeeper");
```

Before starting, you can check which providers are on the classpath:

```java
List<String> types = RegistryFactory.availableTypes();
boolean zookeeperAvailable = RegistryFactory.isAvailable("zookeeper");
```

### 3. Register a service instance

```java
ServiceInstance instance = new ServiceInstance();
instance.setName("game-room");
instance.setId("room-1");
instance.setAddress("127.0.0.1");
instance.setPort(9001);
instance.setWeight(1.0D);
instance.setHealthy(true);
instance.setMetadata(Map.of("zone", "shanghai-a"));

registry.register(instance);
```

`registrationTimeUTC` defaults to the instance creation time; to use a timestamp generated by the business side, override it explicitly via the setter or the builder.

`name`, `address` and a non-blank `id` must not contain leading/trailing whitespace; `name` and a non-blank `id` must not contain `/` either, so the same instance model maps safely onto path-based registries like ZooKeeper and Etcd. When `id` is null or blank, the instance key falls back to `address + ":" + port`, in which case `address` must not contain `/`.

`metadata` accepts only non-`null` string keys/values; writing a null metadata entry via `setMetadata(map)` or the builder's `metadata(key, value)` throws a configuration exception.

Instances can also be created via the builder:

```java
ServiceInstance instance = ServiceInstance.builder()
        .name("game-room")
        .id("room-1")
        .address("127.0.0.1")
        .port(9001)
        .metadata("zone", "shanghai-a")
        .build();
```

### 4. Discover service instances

```java
Collection<ServiceInstance> instances = discovery.getInstances("game-room");

for (ServiceInstance item : instances) {
    System.out.println(item.getAddress() + ":" + item.getPort());
}
```

### 5. Watch for service changes

```java
AutoCloseable handle = discovery.watchService("game-room", event -> {
    ServiceInstance changed = event.getInstance();
    System.out.println(event.getType() + " " + changed.getKey());
});

// close the handle when no longer watching
handle.close();
```

Watching service-name changes:

```java
AutoCloseable handle = discovery.watchServiceNames(event -> {
    System.out.println(event.getType() + " " + event.getServiceName());
});
```

Watch callbacks are invoked directly by the underlying registry's event-dispatch thread, and the initial snapshot is pushed synchronously within the `watchService(...)` / `watchServiceNames(...)` call stack. `onEvent` must not block or throw; for heavy processing, hand off to your own thread group.

## Built-in Types

| Type | `RegistryType` | String key | endpoints example |
| --- | --- | --- | --- |
| ZooKeeper | `RegistryType.ZOOKEEPER` | `zookeeper` | `127.0.0.1:2181` |
| Etcd | `RegistryType.ETCD` | `etcd` | `http://127.0.0.1:2379` |
| Nacos | `RegistryType.NACOS` | `nacos` | `127.0.0.1:8848` |
| Consul | `RegistryType.CONSUL` | `consul` | `127.0.0.1:8500` |

Query snapshots from the built-in providers are stably sorted: instances ascending by instance key, service names in lexicographic order.

## Configuration

| Field | Default | Description |
| --- | --- | --- |
| `type` | none | Registry type; built-in enum or string key |
| `endpoints` | none | Registry address; format varies per provider |
| `basePath` | `/services` | Root path for registration data, used by ZooKeeper and Etcd; leading/trailing whitespace is trimmed and the trailing slash normalized |
| `leaseTtlSeconds` | `10` | Etcd lease TTL |
| `properties` | empty | Provider-private extension settings, passed through or mapped to the corresponding native client |

When setting extension properties via `properties(...)` or `property(key, value)`, keys/values must be strings, `key` must not be blank and `value` must not be `null`; invalid settings throw a configuration exception.

Common provider-private settings:

| Provider | key | Description |
| --- | --- | --- |
| ZooKeeper | `authScheme` / `auth` | Curator auth info, e.g. digest auth |
| ZooKeeper | `connectionTimeoutMillis` / `sessionTimeoutMillis` | Curator connection and session timeouts |
| ZooKeeper | `retryBaseSleepMillis` / `retryMaxRetries` | Curator retry policy |
| Etcd | `user` / `password` | Etcd username/password |
| Etcd | `namespace` / `authority` | jetcd namespace and authority |
| Etcd | `trustedCertPath` | TLS trusted certificate path |
| Etcd | `clientCertPath` / `clientKeyPath` | Mutual-TLS client certificate and private key paths |
| Etcd | `connectTimeoutMillis` / `keepaliveTimeMillis` / `keepaliveTimeoutMillis` | jetcd connection and keepalive parameters |
| Etcd | `keepaliveWithoutCalls` | Whether keepalive without calls is allowed, `true` / `false` |
| Etcd | `retryDelayMillis` / `retryMaxDelayMillis` / `retryMaxAttempts` | jetcd retry parameters |
| Etcd | `maxInboundMessageSize` | gRPC max inbound message size |
| Etcd | `operationTimeoutMillis` | Timeout for synchronous Etcd operations (grant/get/put/delete/revoke etc.), default `5000` |
| Nacos | `serviceNameWatchIntervalMillis` | Service-name watch polling interval, default `5000` |
| Consul | `aclToken` / `token` / `consul.token` | Consul ACL token |
| Consul | `blockingQueryWaitSeconds` / `consul.blockingQueryWaitSeconds` | Consul blocking-query wait time, default `55` |
| Consul | `heartbeatTtlSeconds` / `consul.heartbeatTtlSeconds` | When set, creates a TTL health check per registered instance and heartbeats on a virtual thread; after a process crash the heartbeat stops, the TTL expires, and Consul deregisters the instance after `1m`. Off by default |
| Consul | `watchShutdownTimeoutMillis` / `consul.watchShutdownTimeoutMillis` | Wait time when shutting down the Consul watch thread pool, default `2000` |
| Consul | `pingOnStart` | Whether to call the Consul status API on start as a connectivity check, default `true` |
| Consul | `tags` / `consul.tags` | Default tags for registered instances, comma-separated |
| Consul | `checkHttp` / `checkTcp` / `checkInterval` / `checkTimeout` | Common Consul health-check settings; for complex scenarios build the client or register checks with the native SDK directly |

Boolean settings accept only `true` / `false` (case-insensitive); blank values fall back to the default, anything else throws a configuration exception.

After a keepalive drop, the Etcd provider re-creates the lease and re-registers the instances registered by the current process; when a watch drops or completes, it re-subscribes with exponential backoff and pulls a snapshot before re-subscribing to catch up on changes. Synchronous Etcd remote operations default to a 5-second timeout, adjustable via `operationTimeoutMillis`. For Nacos, `group` / `namespace` are registry-level settings — do not mix them into individual instance metadata; if `__nacos.group` or `__nacos.namespace` disagrees with the registry config, register and deregister fail outright. Nacos instance watch uses the native `subscribe`; service-name watch is implemented by a registry-internal single-thread poller calling `getServicesOfServer` and diffing snapshots, with the interval controlled by `serviceNameWatchIntervalMillis` — closing the watch handle cancels the polling task. The Consul provider takes a single Consul Agent address; watches use the Consul SDK's blocking queries with Consul index change notification rather than fixed-interval HTTP polling. Each watch and each TTL heartbeat runs on its own virtual thread, so concurrent watch count is limited only by Consul itself. The `ConsulRegistry(ConsulClient, Properties)` constructor entry is kept, so business code can still use the native Consul SDK for more complex registration, health-check and ACL scenarios. Note that Consul's "auto-removal on crash" semantics differ from Etcd/ZooKeeper/Nacos: only services registered with a health check get removed by Consul after an instance crashes. Therefore, when any `checkXxx` is configured without an explicit `checkDeregisterCriticalServiceAfter`, it defaults to `1m` so critical instances are deregistered automatically; when no health check is configured at all, registration logs a one-time WARN that crashed instances will not be auto-removed. For auto-removal, set `heartbeatTtlSeconds` (registry-managed TTL heartbeat check, giving Consul the same crash-removal semantics as the other backends), or configure `checkHttp` / `checkTcp`, or register via the native SDK.

## Observability

The built-in implementations expose lightweight read-only state getters for wiring into logging, health checks or metrics:

| Implementation | Observation methods |
| --- | --- |
| `RegistryBundle` | `isStarted()`, `isClosed()` |
| `EtcdRegistry` | `isStarted()`, `isClosed()`, `getRegisteredInstanceCount()`, `getActiveWatchCount()`, `getCurrentLeaseId()`, `isReconnectScheduled()` |
| `NacosRegistry` | `isStarted()`, `isClosed()`, `getActiveWatchCount()`, `getServiceNameWatchIntervalMillis()` |
| `ConsulRegistry` | `isStarted()`, `isClosed()`, `getActiveWatchCount()`, `getWatchShutdownTimeoutMillis()`, `getHeartbeatTtlSeconds()` |
| `ZookeeperRegistry` | `isStarted()`, `isClosed()`, `getActiveWatchCount()` |

## Extending with a Provider

To add a new registry implementation:

1. Implement `Registry` and `Discovery`.
2. Implement `RegistryProvider`, returning a unique string type.
3. List the provider's fully qualified class name in `META-INF/services/com.github.huaye2007.mana.registry.spi.RegistryProvider`.
4. Business code selects the implementation via `config.setType("your-type")`.

Before going to production, complete deployment validation against the target registry (connectivity, lease/session expiry, watch pushes, failover).

## Testing

Run from the repository root:

```bash
mvn "-Dmaven.repo.local=.\.m2" -f game-registry\pom.xml test
```

Notes: the ZooKeeper tests start an embedded server and require loopback networking to be allowed on the machine. The Etcd, Nacos and Consul integration tests are disabled by default; enable them once the local services are available.

Enabling the external integration tests:

```bash
mvn "-Dmaven.repo.local=.\.m2" -f game-registry\pom.xml -Pregistry-it-zookeeper test
mvn "-Dmaven.repo.local=.\.m2" -f game-registry\pom.xml -Pregistry-it-etcd test
mvn "-Dmaven.repo.local=.\.m2" -f game-registry\pom.xml -Pregistry-it-nacos test
mvn "-Dmaven.repo.local=.\.m2" -f game-registry\pom.xml -Pregistry-it-consul test
```

These profiles simply set the corresponding `game.registry.integration.*` system properties to `true`; you can still override with `-D` directly, and `registry-it-all` enables all external integration tests at once.

The Etcd integration tests cover registration/discovery and lease-revoke cleanup on close; the Nacos integration tests cover registration/discovery, instance watch and service-name watch; the Consul integration tests cover instance watch and service-name watch. Etcd defaults to `http://127.0.0.1:2379`, Nacos to `127.0.0.1:8848`, Consul to `127.0.0.1:8500`. The test targets can be overridden with the following system properties:

| System property | Default | Description |
| --- | --- | --- |
| `game.registry.integration.zookeeper` | `false` | ZooKeeper integration test switch |
| `game.registry.integration.etcd.endpoints` | `http://127.0.0.1:2379` | Etcd integration test address |
| `game.registry.integration.etcd.basePath` | `/services` | Etcd integration test root path |
| `game.registry.integration.etcd.ttlSeconds` | `10` | Etcd integration test lease TTL |
| `game.registry.integration.nacos.endpoints` | `127.0.0.1:8848` | Nacos integration test address |
| `game.registry.integration.nacos.group` | `DEFAULT_GROUP` | Nacos integration test group |
| `game.registry.integration.nacos.cluster` | `DEFAULT` | Nacos integration test cluster |
| `game.registry.integration.consul.endpoints` | `127.0.0.1:8500` | Consul integration test address |
